## 二、项目特定规范

### 2.1 技术栈确认

| 层 | 技术 |
|----|------|
| 后端 | Rust (Axum 0.7 + sqlx 0.7 + SQLite) |
| 前端 | React 18 + TypeScript + Vite |
| 样式 | Orange 玻璃拟态风格 |

### 2.2 后端开发规范

**分层架构** (严格遵循):
```
handler.rs    → 路由处理、参数提取、响应构造
service.rs    → 业务逻辑、事务管理
repository.rs → 数据库操作
```

**错误处理**:
- 必须使用 `AppError` 统一错误类型
- 错误响应格式: `{ "code": number, "message": string, "data": null, "request_id": string }`

**数据库**:
- 所有字段默认 `NOT NULL`
- 查询条件字段必须建立索引
- 严禁 N+1 查询

### 2.3 前端开发规范

**样式系统**:
- 使用 CSS 变量，禁止硬编码颜色
- 主色调: Orange (`#f97316`)
- 玻璃拟态效果: `backdrop-blur` + 半透明背景

**组件规范**:
- 原子化设计 (Atoms → Molecules → Organisms)
- 所有交互必须有状态反馈 (Hover/Active/Loading)
- 使用平滑过渡动画

### 2.4 API 设计规范

**响应格式**:
```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "uuid"
}
```

**错误码规范**:
- `0` - 成功
- `400xxx` - 客户端错误 (参数错误、验证失败)
- `401xxx` - 认证错误
- `403xxx` - 权限错误
- `404xxx` - 资源不存在
- `500xxx` - 服务器内部错误

## 三、开发流程

### 3.1 接口先行原则

1. 先定义 API 接口 / TypeScript 类型
2. 提交人类确认
3. 确认后编写实现

### 3.2 无尽审问原则（Zero-Assumption）

**在编写任何代码之前，必须无尽审问用户需求，直到没有任何假设。**

审问内容包括但不限于：
- 多种实现路线的优缺点对比
- 每种方案的应用场景和 trade-offs
- 用户的具体偏好和约束条件
- 未来扩展性考虑

**结束审问的唯一标准**：AI 明确表示"已无任何假设"

### 3.3 代码质量标准

- 函数不超过 20 行
- 嵌套层级不超过 3 层
- 单一职责原则
- 纯函数优先

### 3.4 文档同步

完成任务后必须更新：
- `memories/v0.3-progress.md` - 记录完成的工作
- `memories/PROJECT_STATUS.md` - 更新功能状态
