## 🎯 阶段执行计划

### 阶段4：Backup 模块实现（1.5 天）

#### 4.1 核心需求
- 支持本地备份（保存到 `backups/` 目录）
- 支持 S3 备份（AWS S3 兼容 API）
- 用户创建备份时可选择存储方式
- 自动备份调度（daily / weekly / monthly）
- 恢复时分步骤执行，支持中断恢复

#### 4.2 代码结构
```
src/modules/backup/
├── mod.rs
├── domain.rs         # BackupJob / BackupSchedule / BackupProvider
├── dto.rs
├── repository.rs
├── service.rs        # 核心备份/恢复逻辑
├── scheduler.rs      # 自动调度
└── handler.rs        # HTTP API

src/infra/backup/
├── mod.rs
├── traits.rs         # BackupStorageBackend trait
├── local.rs          # 本地存储实现
└── s3.rs             # S3 存储实现（使用 aws-sdk-s3）
```

#### 4.3 API 端点
- `POST /api/admin/backup` - 创建备份（支持选择存储方式）
- `GET /api/admin/backup/list` - 备份列表
- `POST /api/admin/backup/restore` - 恢复备份
- `GET /api/admin/backup/schedule` - 获取计划
- `PUT /api/admin/backup/schedule` - 更新计划
- `DELETE /api/admin/backup/:id` - 删除备份

#### 4.4 数据库变更
- 新增 `backups` 表（或在 settings 中存储计划）
- 字段：id / created_at / size / storage_backend / status / manifest_hash

#### 4.5 验收标准
- ✅ 能创建本地备份
- ✅ 能创建 S3 备份
- ✅ 能恢复备份
- ✅ 自动备份调度正常工作
- ✅ 架构支持未来扩展其他存储后端

## 📝 关键设计原则

1. **存储后端可插拔**：BackupStorageBackend trait，支持本地 / S3 / 未来扩展
2. **配置统一管理**：所有配置优先存储在 settings 表
3. **向后兼容**：每个阶段都不破坏现有 API
4. **渐进式增强**：每个阶段都能独立发布
5. **用户选择优先**：备份方式、主题选择等让用户决定
