# InkForge 运行期协作规则

> 适用范围：每次新会话、每个新任务、每次准备进入实现之前。
> 目标：让 AI 在本项目中以“先澄清、后执行、按风险分级推进”的方式工作。

---

## 一、每次会话的强制读取顺序

1. `CODEBUDDY.md`
2. `memories/SKILL.md`
3. `memories/PROJECT_STATUS.md`
4. `memories/RUNTIME_RULES.md`

## 二、核心运行规则

### 2.1 先澄清，后执行
- 任何架构方向、Phase 边界、接口策略、迁移方案变更前，必须先提问确认。
- 若仍存在关键假设，不得直接开始实现。
- 结束澄清阶段的唯一标准：AI 必须明确说出 **“已无任何假设”**。

### 2.2 风险分级执行

#### 低风险
可在已授权范围内直接执行，并在自然节点汇报：
- 只读调研
- 当前任务范围内的文档整理
- 可逆的本地低影响改动
- 不改变外部接口语义的轻量维护

#### 中风险
必须先给出变更清单并获得确认：
- 跨多文件的结构性调整
- API 兼容策略调整
- 数据模型扩展
- 迁移脚本设计
- 新增运行规则或治理规则

#### 高风险
必须先单独确认后再执行：
- 删除数据 / 文件 / 分支
- 不可逆 `git` 操作
- 对外发布、推送、评论、通知
- 修改共享基础设施或权限
- 任何可能影响他人或难以回滚的动作

### 2.3 写文件规则
- 已明确授权范围内的文档类写入，可直接执行。
- 代码、配置、迁移、路由、接口契约写入，必须先澄清并确认边界。
- 任何超出当前确认范围的新增文件，必须先说明用途并获确认。

### 2.4 提问触发器
以下情况必须主动提问：
1. 涉及架构方向或阶段顺序变化
2. 涉及高风险或难回滚动作
3. 发现用户目标与当前文档冲突
4. 需要在多个合理方案之间做选择
5. 准备扩展任务范围或新增非显然文件

### 2.5 状态同步
- 多步骤任务必须维护任务列表。
- 一个阶段完成后，必须立即更新进度与状态文档。
- 不允许把多个已完成事项长时间堆积后再一起补记。

## 三、执行协议

### 3.1 标准流程
1. 读取强制文档
2. 复述最小目标
3. 识别假设并逐项提问
4. 明确边界后说明“已无任何假设”
5. 再进入实现 / 写入 / 测试
6. 完成后同步项目文档

### 3.2 范围控制
- 只做用户明确要求的内容。
- 不把“顺手优化”伪装成必要改动。
- 不提前实现远期路线图中的能力。

### 3.3 汇报节奏
- 仅在自然节点汇报：澄清完成、阶段切换、出现 blocker、完成写入。
- 避免对每个微小动作都打断用户。

## 四、联网校验摘要

> 本节用于说明：以上规则并非项目内部自说自话，而是与主流 AI / Agent 运行逻辑相一致。

2026-04-09 已进行联网校验，结论如下：

1. **需求含糊时先澄清** 是主流做法。
   - 外部资料一致支持：当用户意图不清时，应说明假设并提出澄清问题，而不是直接冒进执行。

2. **高影响、难回滚动作前要求人工确认** 是主流做法。
   - 外部资料一致支持：高后果动作应有人类在环；授权范围必须清晰，不能无限放大自主权。

3. **不是每个微小动作都要逐条确认**。
   - 外部资料更支持“风险分级 + 已授权范围内逐步执行”，而不是对每一步都机械确认。

4. **自主权应随证据和范围逐步扩大**。
   - 更合理的模式是：在明确范围内逐步执行；一旦超出范围或进入高风险区间，立即回到确认流程。

- 若未来要修改本文件中的核心规则，必须先重新联网复核，再更新文档。

## 五、与项目其他规则的关系

优先级顺序：
1. 用户当前明确确认的范围
2. `CODEBUDDY.md`
3. `memories/RUNTIME_RULES.md`
4. `memories/SKILL.md`
5. `memories/PROJECT_STATUS.md`

若规则冲突：
- 以更高优先级规则为准；
- 若仍存在歧义，先提问，不得自行假设。
