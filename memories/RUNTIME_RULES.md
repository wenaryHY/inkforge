# InkForge 运行期协作规则

> 适用范围：每次新会话、每个新任务、每次准备进入实现之前。
> 目标：让 AI 在本项目中以"先澄清、后执行、按风险分级推进"的方式工作。

---

## 一、每次会话的强制读取顺序

1. `../CODEBUDDY.md`
2. `memories/PROJECT_STATUS.md`
3. `memories/RUNTIME_RULES.md`

---

## 二、核心运行规则

### 2.1 先澄清，后执行
- 任何架构方向、Phase 边界、接口策略、迁移方案变更前，必须先提问确认。
- 若仍存在关键假设，不得直接开始实现。
- 结束澄清阶段的唯一标准：AI 必须明确说出 **"已无任何假设"**。

### 2.2 风险分级执行

#### 低风险（可直接执行）
- 只读调研
- 当前任务范围内的文档整理
- 可逆的本地低影响改动
- 不改变外部接口语义的轻量维护

#### 中风险（需先确认）
- 跨多文件的结构性调整
- API 兼容策略调整
- 数据模型扩展
- 迁移脚本设计
- 新增运行规则或治理规则

#### 高风险（必须单独确认）
- 删除数据 / 文件 / 分支
- 不可逆 `git` 操作
- 对外发布、推送、评论、通知
- 修改共享基础设施或权限
- 任何可能影响他人或难以回滚的动作

### 2.3 写文件规则
- 已明确授权范围内的文档类写入，可直接执行。
- 代码、配置、迁移、路由、接口契约写入，必须先澄清并确认边界。
- 任何超出当前确认范围的新增文件，必须先说明用途并获确认。

### 2.4 提问触发器
以下情况必须主动提问：
1. 涉及架构方向或阶段顺序变化
2. 涉及高风险或难回滚动作
3. 发现用户目标与当前文档冲突
4. 需要在多个合理方案之间做选择
5. 准备扩展任务范围或新增非显然文件

### 2.5 状态同步
- 多步骤任务必须维护任务列表。
- 一个阶段完成后，必须立即更新进度与状态文档。
- 不允许把多个已完成事项长时间堆积后再一起补记。

## 三、工程质量底线

### 3.1 AI 协作原则（防止屎山代码）

#### 角色契约化
- AI 必须扮演"资深软件架构师/高级工程师"角色。
- 严禁生成未经逻辑推导的代码，在面对模糊需求时必须先提问再执行。

#### 步进式生成逻辑
- **接口先行**：在编写逻辑前，先定义接口（API/Types/Interfaces）并交由人类确认。
- **单元测试对齐**：核心逻辑必须采用 TDD 模式，先生成具体的测试用例，再填充实现。
- **单一职责约束**：每个函数/方法原则上不超过 20 行，嵌套层级严禁超过 3 层。

#### 上下文管理
- 只向 AI 提供必要的上下文，避免因冗余代码导致"幻觉"。
- 每次对话需明确当前遵循的代码风格。

### 3.2 前后端开发规范

#### 前端
- 严谨使用全局 CSS 变量，避免硬编码颜色或间距。
- 遵循 Atomic Design 原则，将 UI 划分为 Atoms, Molecules 和 Organisms。
- 所有交互均需包含交互反馈（Hover, Active, Loading 状态）。

#### 后端
- **防御式编程**：所有外部输入必须经过验证，严禁信任前端传输的任何数据。
- **错误模型**：统一错误处理，确保每个 API 返回结构化的错误码与提示。
- **性能感知**：严查 N+1 查询，所有数据库操作需经过查询计划预判。

#### Rust 模块命名规范（强制）
- **新增模块必须使用 Rust 2018+ 风格**：使用 `directory.rs` + `directory/` 子目录结构，而非传统的 `directory/mod.rs`。
- **示例**：
  - ❌ 旧风格（禁止新增）：`src/utils/mod.rs` + `src/utils/helpers.rs`
  - ✅ 新风格（强制使用）：`src/utils.rs` + `src/utils/helpers.rs`
- **现有代码**：项目中现存 23 个 `mod.rs` 文件使用传统风格，短期不迁移，但所有新增模块必须遵循新规范。
- **理由**：Rust 2018 edition 引入的新风格更简洁，减少文件层级，提升可读性。`mod.rs` 仍然有效但已不推荐用于新代码。

### 3.3 数据库建模规范
- **非空原则**：所有字段默认设为 `NOT NULL`，必要时提供默认值。
- **索引策略**：在频繁作为查询条件的字段上严格建立索引。
- **反规范化决策**：除非面临极限性能瓶颈，否则强制遵循第三范式（3NF）。

### 3.4 算法与性能
- **O(n) 审计**：对所有涉及循环的操作进行时间复杂度评估，严禁产生 O(n²) 以上且无明确说明的逻辑。
- **内存优化**：针对大数据集查询，强制使用流式传输或分页。
- **早期退出机制**：优先处理错误分支与边缘情况，主逻辑保持在函数最外层。

## 四、执行协议

### 4.1 标准流程
1. 读取强制文档
2. 复述最小目标
3. 识别假设并逐项提问
4. 明确边界后说明"已无任何假设"
5. 再进入实现 / 写入 / 测试
6. 完成后同步项目文档

### 4.2 范围控制
- 只做用户明确要求的内容。
- 不把"顺手优化"伪装成必要改动。
- 不提前实现远期路线图中的能力。

### 4.3 汇报节奏
- 仅在自然节点汇报：澄清完成、阶段切换、出现 blocker、完成写入。
- 避免对每个微小动作都打断用户。

---

## 五、文档排版规范

### 5.1 Markdown 规范
- **中英排版**：在中文与西文、数字之间强制添加一个半角空格。
- **层级结构**：严禁跨级设置标题（如 H2 直接跳到 H4）。
- **可视化增强**：尽可能使用 Mermaid 流程图或表格展示复杂逻辑。

### 5.2 Mermaid 绘图标准
- **极简主义**：流向图中节点描述应简洁，避免长句。
- **统一风格**：图表颜色应保持与项目 UI 主色调一致。

1. 用户当前明确确认的范围

- 若仍存在歧义，先提问，不得自行假设。
