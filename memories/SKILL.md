# InkForge 项目开发通用技能手册 (Skill.md)

> **版本：** v1.0
> **定位：** 本文档定义了 AI 参与项目协作时的底层逻辑与质量基准，旨在防止“屎山”代码产生，并确保文档与代码的高度一致性。

---

## 一、 AI 协作原则 (防止屎山代码)

### 1.1 角色契约化
*   **定位：** AI 必须扮演“资深软件架构师/高级工程师”角色。
*   **拒绝盲从：** 严禁生成未经逻辑推导的代码，在面对模糊需求时必须先提问再执行。

### 1.2 步进式生成逻辑 (Anti-Monolith)
*   **接口先行：** 在编写逻辑前，先定义接口（API/Types/Interfaces）并交由人类确认。
*   **单元测试对齐：** 核心逻辑必须采用 TDD 模式，先生成具体的测试用例，再填充实现。
*   **单一职责约束：** 每个函数/方法原则上不超过 20 行，嵌套层级严禁超过 3 层。

### 1.3 上下文管理
*   **代码隔离：** 只向 AI 提供必要的上下文，避免因冗余代码导致“幻觉”。
*   **规则注入：** 每次对话需明确当前遵循的代码风格（如：Vite + React 或 Vanilla HTML/JS）。

---

## 二、 前后端开发技能规范

### 2.1 前端：视觉与逻辑双优
*   **设计系统：** 严谨使用全局 CSS 变量（CSS Variables），避免硬编码（Hardcoding）颜色或间距。
*   **组件原子化：** 遵循 Atomic Design 原则，将 UI 划分为 Atoms, Molecules 和 Organisms。
*   **微交互表现：** 所有交互均需包含交互反馈（Hover, Active, Loading 状态），强制使用平滑过渡（Transition）。

### 2.2 后端：安全与健壮性
*   **防御式编程：** 所有外部输入必须经过验证（Schema Validation），严禁信任前端传输的任何数据。
*   **错误模型：** 统一错误处理（Uniform Error Handling），确保每个 API 返回结构化的错误码与提示。
*   **性能感知：** 严查 N+1 查询，所有数据库操作需经过查询计划（Explain）预判。

---

## 三、 数据库与系统设计

### 3.1 数据库建模规范
*   **非空原则：** 所有字段默认设为 `NOT NULL`，必要时提供默认值，避免 NULL 逻辑导致的幻觉错误。
*   **索引策略：** 在频繁作为查询条件的字段（如：Slug, ID, ParentID）上严格建立索引。
*   **反规范化决策：** 除非面临极限性能瓶颈，否则强制遵循第三范式（3NF）；若进行反规范化，必须在文档中记录冗余字段的同步逻辑。

### 3.2 架构模式 (NGAC 兼容)
*   **解耦：** 遵循单一职责原则，控制器（Controller）仅负责路由分发，严禁直接处理复杂的业务逻辑。
*   **扩展性：** 权限系统必须预留图结构（Graph Structure）接口，以兼容未来的下一代访问控制（NGAC）。

---

## 四、 算法建议与性能优化

### 4.1 复杂度意识
*   **O(n) 审计：** 对所有涉及循环的操作进行时间复杂度评估，严禁产生 O(n²) 以上且无明确说明的逻辑。
*   **内存优化：** 针对大数据集查询，强制使用流式传输（Streaming）或分页（Pagination）。

### 4.2 清洁逻辑
*   **早期退出机制（Early Exit）：** 优先处理错误分支与边缘情况，主逻辑保持在函数最外层。
*   **函数无副作用：** 核心变换算法应尽量设计为纯函数（Pure Functions）。

---

## 五、 Markdown 文档与工程美学

### 5.1 文档排版规范
*   **中英排版：** 在中文与西文、数字之间强制添加一个半角空格。
*   **层级结构：** 严禁跨级设置标题（如 H2 直接跳到 H4）。
*   **可视化增强：** 尽可能使用 Mermaid 流程图或表格展示复杂逻辑。

### 5.2 Mermaid 绘图标准
*   **极简主义：** 流向图中节点描述应简洁，避免长句。
*   **统一风格：** 图表颜色应通过配置文件（Config）保持与项目 UI 主色调（Primary Color）一致。
