# InkForge 可执行 Roadmap

> 生成日期：2026-04-09
> 目标：把架构愿景改写成可落地、可验收、可分阶段推进的执行路线。
> 方法：以当前仓库真实状态为基线，先做低风险基础演进，再做安装向导、桌面集成、插件和生态。

---

## 当前基线

### 已有基础
- Rust Core + Axum + SQLite 已稳定运行。
- React 管理后台已可用。
- MiniJinja 前台主题渲染已跑通。
- 备份、回收站、SEO、WebSocket 已接入主链路。

### 当前短板
- API 仍以 `/api/...` 为主，尚未完成版本化。
- 后台编辑器仍以 Milkdown 为主，维护成本偏高。
- 内容模型仍以单一 post 为核心，尚无 `content_type`。
- 安装向导、Tauri、插件运行时仍未落地。

---

## 总体执行原则

1. **接口先行**：先定义 migration、DTO、TS types、路由契约，再进入实现。
2. **最小可用优先**：每个阶段只解决主链路，不把远期能力提前塞进当前阶段。
3. **兼容平滑迁移**：涉及 API、数据模型、编辑器替换时，优先保留过渡层。
4. **风险分级推进**：低风险改动可直接执行；架构、接口策略、共享状态变更先确认。
5. **文档同步**：每个阶段完成后同步更新 `memories/v0.3-progress.md` 与 `memories/PROJECT_STATUS.md`。

## Phase 1 — 内容底座升级

### 目标
先稳定内容编辑与内容模型，给后续安装向导、主题构建和插件扩展提供统一基础。

### 范围
1. Milkdown → Tiptap `MVP`
2. `posts` 增加 `content_type`
3. 引入 `/api/v1/...` 兼容层，但暂不立即移除旧 `/api/...`

### 交付物
- Tiptap 编辑器最小可用版
- `post | page | portfolio` 三类内容模型
- 后台按内容类型创建、编辑、筛选
- 后端接口支持按 `content_type` 工作
- `/api/v1` 路由可用，旧路径保留过渡期兼容

### 明确不做
- 不做修订历史
- 不做 tantivy 搜索迁移
- 不做复杂 block schema
- 不做 portfolio 专属扩展表

### 退出条件
- 三类内容可完成创建、编辑、发布、读取
- 旧文章数据可无损读取
- 媒体插入链路在新编辑器下保持可用
- 新旧 API 路径都可访问

### 依赖
无，作为后续所有阶段前置。

---

## Phase 2 — 安装向导（Web 先行）

### 目标
让系统具备首次运行初始化能力，先在 Web 路径完成完整安装闭环。

### 范围
1. 安装状态检测
2. setup service / repository / handler
3. Web 安装向导 UI
4. 安装完成后的运行态切换

### 交付物
- 已安装 / 未安装 / 安装中 状态判断机制
- 初始化管理员、站点信息、基础设置的服务层
- Web 安装向导页面
- 首次安装后自动进入正常后台流程

### 明确不做
- 不做 Tauri 原生安装 UI
- 不做云同步
- 不做多租户初始化

### 退出条件
- 空库状态下可仅通过 Web 完成首装
- 安装完成后可直接登录后台
- 重启后不会重复进入安装页
- 安装逻辑与 UI 解耦，可被 Tauri 复用

### 依赖
依赖 Phase 1 的内容底座与接口约束稳定。

---

## Phase 3 — Tauri 基座与原生引导

### 目标
建立桌面应用基座，并复用 Phase 2 的安装逻辑，而不是再做一套新后端逻辑。

### 范围
1. `src-tauri` 基础工程
2. 桌面启动与 Web 后台接入
3. 原生引导 UI
4. 安装逻辑共享层接入

### 交付物
- Tauri 工程骨架
- 桌面端开发与运行链路
- 未安装状态下的原生引导入口
- 安装完成后进入正常管理界面

### 明确不做
- 不做自动更新
- 不做双窗口 / 侧边栏模式切换
- 不做复杂原生能力桥接

### 退出条件
- 桌面端能启动到后台
- 未安装时进入原生引导
- 安装完成后进入正常运行态
- Web 与 Tauri 共用同一套 setup 逻辑

### 依赖
依赖 Phase 2 的安装状态与 setup service 稳定。

---

## Phase 4 — 插件系统 MVP

### 目标
先做可运行插件最小闭环，不在 `MVP` 阶段同时推进双运行时和复杂平台能力。

### 范围
1. 仅做 JS sidecar 单轨
2. manifest / 扫描 / 启停
3. 基础 Slot 注入
4. 基础 Route 注册
5. 最小权限声明

### 交付物
- 插件目录规范
- manifest 结构
- 插件扫描与加载器
- sidecar 启停与状态管理
- MessagePack `IPC MVP`
- 后台最少一个可运行 demo 插件

### 明确不做
- 不做 WASM 插件
- 不做服务注册体系
- 不做复杂插件市场
- 不做全自动拦截式安全扫描

### 退出条件
- 至少一个 demo 插件可安装、启用、停用
- 插件可注册一个页面或一个 Slot
- 插件崩溃不会拖垮主进程
- 权限声明可被安装 / 启用流程识别

### 依赖
依赖 Phase 3 的桌面 / 运行壳能力稳定。

---

## Phase 5 — 主题构建与多端发布

### 目标
把主题开发、构建、数据注入和发布串成一条最小工具链。

### 范围
1. 主题构建产物规范
2. `<script id="ink-data">` 数据注入协议
3. `ink build MVP`
4. 首批发布适配器

### 交付物
- 主题输入 / 输出目录规范
- `ink-data` 数据结构协议
- `ink build` 命令
- 至少一种静态发布目标
- 与现有 MiniJinja SSR 的共存策略

### 明确不做
- 不做完整主题市场
- 不做所有部署平台一次覆盖
- 不做复杂增量发布

### 退出条件
- 示例主题可通过 `ink build` 产出静态资源
- 构建产物可读取 `ink-data`
- 至少一个发布目标可用
- 不破坏现有 MiniJinja 主题运行链路

### 依赖
依赖 Phase 1 的内容模型、Phase 4 的插件扩展边界稳定。

---

## Phase 6 — 生态基建

### 目标
补足最基础的分发、扫描、权限治理文档与设施。

### 范围
1. Registry 静态 `JSON index`
2. 插件安全扫描与风险标签
3. 权限模型升级预留

### 交付物
- Registry 索引格式
- 插件 / 主题包元数据规范
- 风险标签展示链路
- 权限声明到 UI 展示链路
- 为未来 `NGAC` 预留数据模型接口

### 明确不做
- 不立即完整落地 `NGAC`
- 不做官方在线生态服务
- 不做商业化市场能力

### 退出条件
- 插件 / 主题可被索引与发现
- 插件包可展示风险报告
- 权限声明可被安装和启用流程消费
- `NGAC` 仅保留抽象，不强行提前落地

### 依赖
依赖 Phase 4 的插件包格式稳定、Phase 5 的构建产物规范稳定。

---

## 推荐推进顺序

```text
Phase 1 内容底座升级
   ↓
Phase 2 安装向导（Web 先行）
   ↓
Phase 3 Tauri 基座与原生引导
   ↓
Phase 4 插件系统 MVP
   ↓
Phase 5 主题构建与多端发布
   ↓
Phase 6 生态基建
```
