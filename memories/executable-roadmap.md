# InkForge 可执行路线图

> 目标：在不留明显技术债的前提下，按阶段把 InkForge 从稳定 Web 单体演进到稳定桌面壳与可扩展平台。
> 原则：**分阶段推进，不等于用 MVP 借口留下基础设施缺口。**

---

## 当前基线

### 已完成的基础
- Rust + Axum + SQLite 主链路稳定存在
- React 管理后台、Tiptap 编辑器、API v1 兼容层已落地
- MiniJinja 前台主题渲染已落地
- 安装向导 Web 闭环已落地
- 备份、回收站、SEO、WebSocket 已接入主链路

### 当前短板
- 桌面壳 `src-tauri/` 已出现，但未通过构建与打包验收
- 安全基线未补齐
- 插件系统仍是接口占位
- 主题构建工具链、SDK、生态治理均未开始
- 文档与验收机制刚刚收敛，仍需执行化

---

## 总体执行原则

1. **地基优先**：先补齐运行壳、安全、构建、验收边界，再扩展生态。
2. **接口先行**：任何新增阶段都先定义契约、边界和退出条件。
3. **验收先于记账**：通过构建、主链路验证、文档同步后，阶段才算完成。
4. **单一事实源**：状态只写进 `PROJECT_STATUS.md`，进度只写进 `v0.3-progress.md`。
5. **收敛边界再扩张**：在桌面壳和权限模型未稳定前，不前推插件生态。

---

## Phase 1 — 内容底座升级（已完成）

### 已完成结果
- Tiptap 替换 Milkdown
- `post | page` 内容类型接入
- `page_render_mode` 接入
- `/api/v1` 正式路由 + legacy 兼容层
- 后台按内容类型创建、编辑、筛选

### 当前结论
本阶段已完成，后续只做回归与基线修复，不再作为当前主攻方向。

---

## Phase 2 — 安装向导（Web 先行，已完成）

### 已完成结果
- setup 状态模型与旧库回填
- setup service / repository / handler
- `/setup` Web 安装向导 UI
- 安装完成后的运行态切换
- 安装逻辑与 UI 解耦，可供桌面壳复用

### 当前结论
本阶段已完成，未来桌面端必须复用这套逻辑，而不是再造安装后端。

---

## Phase 3A — 基线加固与验收补齐（当前阶段）

### 目标
把当前仓库从“功能很多但边界不够硬”收敛到“构建可靠、安全基线清晰、阶段验收可执行”的状态，为桌面壳和未来插件系统打地基。

### 范围
1. 修复 workspace / 前端 / Tauri 相关构建链路
2. 补齐当前主链路的安全基线
3. 建立阶段性集成测试与验收门槛
4. 清理状态文档与进度文档漂移

### 交付物
- `cargo check --workspace` 可通过
- `npm run build --prefix src/admin/ui` 可通过
- 安全基线整改清单与实现落地
- 更新后的集成测试清单与阶段勾选清单
- 文档与代码状态重新一致

### 明确不做
- 不在这一阶段推进插件系统
- 不在这一阶段推进主题构建 CLI
- 不在这一阶段推进自动更新
- 不提前拆 `inkforge-core`

### 退出条件
- Rust workspace 构建通过
- 前端生产构建通过
- 现有 Web 安装 / 登录 / 管理后台主链路可回归
- 安全高优先项已修完或有明确阻断说明
- `PROJECT_STATUS.md` 与 `v0.3-progress.md` 无失真内容

### 依赖
无；这是桌面壳和生态能力之前的前置阶段。

---

## Phase 3B — Tauri 运行壳落地

### 目标
让桌面端真正成为可运行、可打包、可验收的分发壳，而不是仅有 `src-tauri/` 代码骨架。

### 范围
1. 明确并落实 Tauri 启动模型
2. 打通桌面开发与打包链路
3. 让桌面端复用 Web setup 逻辑
4. 完成桌面态下 `/setup` → `/admin` 运行闭环
5. 处理 sidecar / 窗口 / 生命周期 / 资源打包问题

### 交付物
- 可工作的 `src-tauri/` 工程
- 通过验证的 sidecar 或替代启动模型
- setup / admin 的桌面切换闭环
- sidecar 资源与目标三元组打包方案
- 退出时的进程清理与异常恢复策略

### 明确不做
- 不做自动更新
- 不做复杂原生能力桥接
- 不做插件系统桥接
- 不默认引入多窗口，除非有明确必要性与验收依据

### 退出条件
- `tauri dev` 可稳定启动
- 打包产物可运行
- 未安装状态进入 setup
- 已安装状态进入 admin
- 不出现孤儿进程
- Web 与桌面共用同一套 setup 逻辑

### 依赖
依赖 `Phase 3A` 已完成。

---

## Phase 4 — 插件运行时基础版

### 目标
在运行壳稳定后，建立一个可扩展但不失控的插件运行时基础层。

### 范围
1. manifest 结构
2. 插件扫描与启停
3. 生命周期与隔离边界
4. Slot / Route 注册契约
5. 权限声明最小模型
6. 至少一个 demo 插件

### 交付物
- 插件目录规范
- manifest 协议
- 插件扫描与状态管理
- 生命周期管理与崩溃隔离
- 后台最少一个可运行 demo 插件

### 明确不做
- 不同时推进双轨插件运行时
- 不做插件市场
- 不做复杂服务注册体系
- 不做自动化代码扫描平台

### 退出条件
- demo 插件可安装、启用、停用
- 插件可注册一个页面或一个 Slot
- 插件崩溃不拖垮主进程
- 权限声明可被安装 / 启用流程消费

### 依赖
依赖 `Phase 3B` 稳定。

---

## Phase 5 — 主题构建与发布基础版

### 目标
把主题开发、构建、数据注入和发布适配器串成一条稳定工具链。

### 范围
1. 主题输入 / 输出目录规范
2. `ink-data` 数据注入协议
3. 构建命令入口
4. 至少一个发布适配器
5. 与现有 MiniJinja SSR 的共存边界

### 退出条件
- 示例主题可构建
- 构建产物可读取注入数据
- 至少一个发布目标可用
- 不破坏现有 MiniJinja 运行链路

### 依赖
依赖 `Phase 4` 的扩展边界稳定。

---

## Phase 6 — 生态治理与权限模型预留

### 目标
为未来的主题 / 插件生态提供可治理、可发现、可解释的基础设施。

### 范围
1. Registry 索引格式
2. 插件 / 主题元数据规范
3. 风险标签与权限展示
4. 高级权限模型预留接口

### 退出条件
- 插件 / 主题可被索引与发现
- 风险标签可展示
- 权限声明可被安装流程消费
- 后续高级权限模型有明确落点但不阻塞当前系统

### 依赖
依赖 `Phase 4` 与 `Phase 5` 的格式稳定。

---

## 推荐推进顺序

```text
Phase 1 内容底座升级（已完成）
   ↓
Phase 2 安装向导（Web 先行，已完成）
   ↓
Phase 3A 基线加固与验收补齐（当前）
   ↓
Phase 3B Tauri 运行壳落地
   ↓
Phase 4 插件运行时基础版
   ↓
Phase 5 主题构建与发布基础版
   ↓
Phase 6 生态治理与权限模型预留
```

---

## 当前阶段的现实提醒

如果 `Phase 3A` 没做完，就直接推进插件系统或主题构建链，后续会同时承担：
- 运行壳不稳定
- 安全基线缺失
- 文档与实现继续漂移
- 验收闭环继续失真

因此，**当前真正的下一阶段不是“做更多功能”，而是“让现有基础设施达到可承载后续复杂度的程度”**。
