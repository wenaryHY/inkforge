# InkForge v0.3 阶段6A - 主题管理闭环实施计划

**创建时间**: 2026-04-06
**状态**: 规划中 → 实施中
**优先级**: P0 / 立即开始
**预计工作量**: 3-4 天

---

## 概述

阶段6A 是对原"阶段6 前端 UI 集成"的重新定位。基于当前仓库现状分析：

- Media 分类 UI 已在阶段5-6 期间部分完成（MediaCategories.tsx / hooks / components 已存在）
- 主题模块是最接近长期路线、但尚未闭环的能力
- 主题系统直接连接：Web-Based Theme Engine → Tauri 承载 → 第三方开发者生态

因此，**阶段6A 的核心目标是把主题模块从"可列出/可激活"升级到"可查看详情 / 可编辑配置 / 可上传 / 可验证"的完整后台管理能力**。

---

## 当前主题模块现状评估

### ✅ 已具备
- 后端主题扫描 (`ThemeService::scan_themes`)
- 主题激活 (`ThemeService::activate_theme`)
- 主题列表 API (`GET /api/admin/themes`)
- 主题激活 API (`POST /api/admin/themes/:slug/activate`)
- 前端主题列表页 (`Themes.tsx`)
- 主题 manifest 结构定义

### ❌ 缺失
- 主题详情 API（manifest + 当前配置）
- 主题配置保存 API（虽然 service 层已有 `save_theme_config`，但未接到 router）
- 主题上传 API 的完整校验与错误处理
- 前端主题详情/配置编辑页面
- Schema 驱动的动态配置表单
- 主题预览功能

### ⚠️ 需要明确的边界
- 主题包上传时的冲突处理策略
- manifest 解析失败的错误返回规范
- 配置 schema 字段类型的完整列表
- 前台主题渲染链路（仍是 TODO）

## 迭代一：主题管理闭环（Theme Admin Closure）

### 目标
完成主题后台管理的完整 CRUD 能力，为迭代二的前台运行链路打下基础。

### 任务分解

#### 任务 1.1：补齐主题 API 契约

**后端工作**：

1. **新增 `GET /api/admin/themes/:slug/detail`**
   - 返回：manifest + 当前配置 + schema
   - 用于前端详情页加载

2. **新增 `PATCH /api/admin/themes/:slug/config`**
   - 接收：配置对象
   - 保存到数据库
   - 返回：保存后的配置

3. **完善 `POST /api/admin/themes/upload`**
   - 校验：必须包含 `theme.toml`
   - 校验：manifest 解析成功
   - 处理：slug 冲突（覆盖 vs 拒绝）
   - 返回：统一的上传结果结构

**前端工作**：

1. 在 `types/index.ts` 中补齐 DTO：
   ```typescript
   export interface ThemeDetail {
     manifest: ThemeManifest;
     config: Record<string, any>;
     schema: Record<string, ThemeConfigField>;
   }

   export interface SaveThemeConfigRequest {
     config: Record<string, any>;
   }

   export interface ThemeUploadResponse {
     slug: string;
     name: string;
     version: string;
     message: string;
   }
   ```

2. 在 `services/themeService.ts` 中新增：
   ```typescript
   export async function getThemeDetail(slug: string): Promise<ThemeDetail>
   export async function saveThemeConfig(slug: string, config: Record<string, any>): Promise<void>
   export async function uploadTheme(file: File): Promise<ThemeUploadResponse>
   ```

**预计工作量**: 2-3 小时

#### 任务 1.3：实现 Schema 驱动配置表单

**目标**：从 `ThemeManifest.config` 读取字段定义，自动生成表单。

**支持的字段类型**：
- `text`: 文本输入
- `number`: 数字输入
- `color`: 颜色选择器
- `select`: 下拉选择
- `boolean`: 复选框

**实现步骤**：

1. **定义 schema 字段类型**
   ```typescript
   export interface ThemeConfigField {
     type: 'text' | 'number' | 'color' | 'select' | 'boolean';
     label: string;
     description?: string;
     default?: any;
     required?: boolean;
     options?: Array<{ label: string; value: any }>; // for select
     min?: number; // for number
     max?: number; // for number
   }
   ```

2. **创建 `components/theme/ConfigFieldRenderer.tsx`**
   - 根据字段类型渲染对应的输入组件
   - 处理值的绑定与变更

3. **在 `ThemeConfigForm.tsx` 中使用**
   - 遍历 schema 字段
   - 为每个字段渲染对应的输入组件
   - 收集表单值并提交

**预计工作量**: 2-3 小时

---

#### 任务 1.4：明确上传包校验边界

**后端工作**：

在 `src/modules/theme/handler.rs` 的 `upload_theme_archive` 中补齐：

1. **解析 zip 包**
   - 检查是否包含 `theme.toml`
   - 提取 manifest

2. **验证 manifest**
   - 必填字段：name, slug, version
   - 可选字段：description, author, min_inkforge_version, config

3. **处理 slug 冲突**
   - 策略：允许覆盖（新版本替换旧版本）
   - 返回：`{ success: true, slug, name, version, message: "主题已上传/更新" }`

4. **错误处理**
   - 无效 zip：`{ code: 400, message: "无效的主题包" }`
   - 缺少 theme.toml：`{ code: 400, message: "主题包缺少 theme.toml" }`
   - manifest 解析失败：`{ code: 400, message: "theme.toml 格式错误: ..." }`

**前端工作**：

在 `ThemeUploadDialog.tsx` 中：
- 显示上传进度
- 根据错误码显示对应提示
- 上传成功后刷新列表

**预计工作量**: 1-2 小时

## 迭代二：前台主题运行链路打通（Theme Runtime Bridge）

### 目标
让"激活主题"不再只是后台状态变化，而是能真正影响前台展示，形成最小可运行的 Theme Engine 雏形。

### 任务分解

#### 任务 2.1：落地前台主题渲染入口

**后端工作**：

在 `src/bootstrap/router.rs` 中启用前台主题渲染路由：

```rust
.route("/", get(modules::theme::handler::render_home))
.route("/posts/:slug", get(modules::theme::handler::render_post))
.route("/profile", get(modules::user::theme_handler::render_profile))
.route("/static/*path", get(modules::theme::handler::serve_active_static))
```

实现这些 handler：
- `render_home`: 渲染首页（posts list）
- `render_post`: 渲染文章详情页
- `render_profile`: 渲染用户资料页
- `serve_active_static`: 提供激活主题的静态资源

**预计工作量**: 2-3 小时

## 关键设计原则

1. **最小化前置依赖**
   - 不做 Tauri 集成
   - 不做复杂 schema 引擎
   - 只做 Web 环境下的完整闭环

2. **为长期路线预留接口**
   - Schema 字段类型可扩展
   - 主题配置存储方式可迁移
   - 前台渲染链路可抽象

3. **渐进式增强**
   - 迭代一完成后可独立发布
   - 迭代二完成后可独立发布
   - 每个迭代都能增加真实价值
