# InkForge 架构决策记录

> 目的：只记录已经接受、或明确进入待复核队列的架构决策。
> 规则：没有经过实现验证或明确确认的内容，不得写成既定结论。

---

## 状态说明

- **已接受**：当前执行应遵循的正式决策。
- **待复核**：方向已提出，但尚未通过实现、验收或再次确认。
- **已延后**：确认不在当前阶段推进。

---

## 已接受决策

### AD-001：以 Rust 单体内核作为当前唯一业务真相源
- **状态**：已接受
- **结论**：当前阶段继续以 `Axum + SQLite + React Admin` 的单体结构作为主系统，不提前拆 `inkforge-core`、不做多运行时并行实现。
- **原因**：现有业务能力都落在这条主链路上；在测试、验收和桌面壳稳定前，过早拆核只会放大维护成本。

### AD-002：安装逻辑必须保持单一来源
- **状态**：已接受
- **结论**：安装状态、初始化管理员、站点配置写入，都必须继续由 `setup service / repository / handler` 负责；Web 与未来桌面端只能复用这套逻辑，不能再造第二套安装后端。
- **原因**：安装向导已在 Web 端形成完整闭环，这是未来桌面壳最值得复用的边界。

### AD-003：`/api/v1` 是正式契约，旧 `/api` 仅作兼容层
- **状态**：已接受
- **结论**：所有新增调用一律以 `/api/v1/...` 为准；旧路径只作为过渡兼容，不再扩展新语义。
- **原因**：当前代码已经形成 `v1 + legacy` 双层，继续摇摆只会扩大维护面。

### AD-004：主链路风险收敛优先于新功能扩张
- **状态**：已接受
- **结论**：在进入插件系统、主题构建链和生态能力前，必须先补齐当前的主链路风险与验收闭环。
- **当前优先项**：
  - setup / register 边界必须闭合
  - 后台路由必须保持 `/admin/*` 语义一致
  - 设置保存链路必须与 UI 能力一致
  - 备份、部署与静态入口路径必须统一
  - Markdown 输出与剩余安全高优先项必须补齐
- **原因**：这些问题都已经位于现有主链路上，继续叠加功能只会扩大未来返工成本。

### AD-005：阶段完成以验收通过为准，不以“代码已写”记账
- **状态**：已接受
- **结论**：构建失败、打包未闭环、资源未准备完成的阶段，不能记为完成；进度文档只能反映通过验证的结果。
- **原因**：先记账再验收会直接误导后续路线和优先级判断。

### AD-006：插件系统在运行壳稳定前不得前推
- **状态**：已接受
- **结论**：插件系统继续保持“接口预留”状态，直到桌面运行壳、权限模型、进程生命周期和验收基线明确后再推进。
- **原因**：插件系统会放大运行时边界、权限、安全和稳定性问题，不适合建立在未收敛的壳层之上。

### AD-007：`memories/README.md` 只做导航，不再承载状态结论
- **状态**：已接受
- **结论**：`README.md` 只保留文档索引、读取顺序与职责边界；当前状态只能写进 `PROJECT_STATUS.md`，审计结论只能写进独立审计文档。
- **原因**：导航文档与状态文档混写，是 `memories/` 变得难以快速定位的直接原因之一。

## 待复核决策

### ADR-P01：Tauri 启动模型是否最终采用 sidecar
- **状态**：待复核
- **当前判断**：sidecar 具备较强隔离性，也能复用现有 Axum 服务；但它会引入打包、目标三元组、资源复制、进程回收和 dev / prod 一致性成本。
- **复核标准**：
  1. `cargo check --workspace` 通过
  2. `npm run build --prefix src/admin/ui` 通过
  3. `tauri dev` 可稳定启动
  4. 打包产物能正确携带 sidecar 二进制
  5. 退出后无孤儿进程
- **说明**：在这些条件满足前，sidecar 只能算“当前倾向”，不能算最终定论。

### ADR-P02：桌面窗口模型默认应优先单窗口
- **状态**：待复核
- **当前判断**：除非有明确的窗口尺寸、生命周期或系统能力需求，否则默认应优先单窗口内完成 `/setup` → `/admin` 导航；多窗口应视为经过论证的特例，而不是默认方案。
- **原因**：多窗口会扩大状态同步和进程回收复杂度，不符合当前“先稳后扩”的方向。

### ADR-P03：`inkforge-core` 拆分延后到第二运行时真正出现之后
- **状态**：待复核
- **当前判断**：只有当 Axum 主程序与桌面壳真正共享大量启动 / 服务 / 领域逻辑、并且单体入口已明显成为维护瓶颈时，才启动核心库拆分。
- **原因**：当前更大的问题不是复用率不够，而是边界未稳定、验收未闭环。

## 已延后事项

- 双轨插件运行时（WASM + JS sidecar）
- 自动更新
- `window.cms.*` JavaScript SDK
- tantivy 搜索迁移
- NGAC 的实体化落地
- Registry / 风险扫描 / 生态市场

这些方向会继续保留在已延后决策与路线图中，但**不属于当前执行承诺**。
