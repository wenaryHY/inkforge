# InkForge 架构决策记录

> 目的：记录已接受的架构决策和可执行路线图。
> 规则：没有经过实现验证或明确确认的内容，不得写成既定结论。

## 已接受决策

### AD-001：以 Rust 单体内核作为当前唯一业务真相源
- **状态**：已接受
- **结论**：当前阶段继续以 `Axum + SQLite + React Admin` 的单体结构作为主系统，不提前拆 `inkforge-core`、不做多运行时并行实现。
- **原因**：现有业务能力都落在这条主链路上；在测试、验收和桌面壳稳定前，过早拆核只会放大维护成本。

### AD-002：安装逻辑必须保持单一来源
- **状态**：已接受
- **结论**：安装状态、初始化管理员、站点配置写入，都必须继续由 `setup service / repository / handler` 负责；Web 与未来桌面端只能复用这套逻辑，不能再造第二套安装后端。
- **原因**：安装向导已在 Web 端形成完整闭环，这是未来桌面壳最值得复用的边界。

### AD-003：`/api/v1` 是正式契约，旧 `/api` 仅作兼容层
- **状态**：已接受
- **结论**：所有新增调用一律以 `/api/v1/...` 为准；旧路径只作为过渡兼容，不再扩展新语义。
- **原因**：当前代码已经形成 `v1 + legacy` 双层，继续摇摆只会扩大维护面。

### AD-004：主链路风险收敛优先于新功能扩张
- **状态**：已接受
- **结论**：在进入插件系统、主题构建链和生态能力前，必须先补齐当前的主链路风险与验收闭环。
- **原因**：这些问题都已经位于现有主链路上，继续叠加功能只会扩大未来返工成本。

### AD-005：阶段完成以验收通过为准
- **状态**：已接受
- **结论**：构建失败、打包未闭环、资源未准备完成的阶段，不能记为完成；进度文档只能反映通过验证的结果。
- **原因**：先记账再验收会直接误导后续路线和优先级判断。

### AD-006：插件系统在运行壳稳定前不得前推
- **状态**：已接受
- **结论**：插件系统继续保持"接口预留"状态，直到桌面运行壳、权限模型、进程生命周期和验收基线明确后再推进。
- **原因**：插件系统会放大运行时边界、权限、安全和稳定性问题，不适合建立在未收敛的壳层之上。

### AD-007：文档职责边界必须清晰
- **状态**：已接受
- **结论**：`PROJECT_STATUS.md` 只承载当前状态结论；`architecture-decisions.md` 只记录架构决策与路线图；`RUNTIME_RULES.md` 只记录运行规则与工程质量底线。
- **原因**：导航文档与状态文档混写，是文档变得难以快速定位的直接原因。

## 待复核决策

### ADR-P01：Tauri 启动模型是否最终采用 sidecar
- **状态**：待复核
- **当前判断**：sidecar 具备较强隔离性，也能复用现有 Axum 服务；但它会引入打包、目标三元组、资源复制、进程回收和 dev / prod 一致性成本。
- **复核标准**：
  1. `cargo check --workspace` 通过
  2. `npm run build --prefix src/admin/ui` 通过
  3. `tauri dev` 可稳定启动
  4. 打包产物能正确携带 sidecar 二进制
  5. 退出后无孤儿进程

### ADR-P02：桌面窗口模型默认应优先单窗口
- **状态**：待复核
- **当前判断**：除非有明确的窗口尺寸、生命周期或系统能力需求，否则默认应优先单窗口内完成 `/setup` → `/admin` 导航。

### ADR-P03：`inkforge-core` 拆分延后到第二运行时真正出现之后
- **状态**：待复核
- **当前判断**：只有当 Axum 主程序与桌面壳真正共享大量启动 / 服务 / 领域逻辑、并且单体入口已明显成为维护瓶颈时，才启动核心库拆分。

## 可执行路线图

### Phase 1 — 内容底座升级（已完成）
- Tiptap 替换 Milkdown
- `post | page` 内容类型接入
- `/api/v1` 正式路由 + legacy 兼容层

### Phase 2 — 安装向导（已完成）
- setup 状态模型与旧库回填
- setup service / repository / handler
- `/setup` Web 安装向导 UI
- 安装完成后的运行态切换

### Phase 3A — 基线加固与验收补齐（当前阶段）

**目标**：把当前仓库从"功能很多但边界不够硬"收敛到"构建可靠、安全基线清晰、阶段验收可执行"的状态。

**已完成项**：
- [x] 删除旧迁移入口残留（`src/db.rs`、`src/infra/db/pool.rs`）
- [x] 添加编译优化配置（`lto`、`opt-level`）
- [x] 建立最小 CI

**待完成项**：
- [ ] 摘除 legacy `/api` 兼容层并补最小回归验收
- [ ] 启用 Tauri 打包配置

**退出条件**：
- Rust workspace 构建通过 ✅
- 前端生产构建通过 ✅
- Web 主链路回归通过
- 文档与实际状态一致

### Phase 3B — Tauri 运行壳落地

**目标**：让桌面端成为可运行、可打包、可验收的分发壳。

**交付物**：
- 可工作的 `src-tauri/` 工程
- 通过验证的 sidecar 或替代启动模型
- setup / admin 的桌面切换闭环

**退出条件**：
- `tauri dev` 可稳定启动
- 打包产物可运行
- 不出现孤儿进程

### Phase 4 — 插件运行时基础版
- manifest 结构
- 插件扫描与启停
- 生命周期与隔离边界
- 至少一个 demo 插件

### Phase 5 — 主题构建与发布基础版
- 主题输入 / 输出目录规范
- `ink-data` 数据注入协议
- 构建命令入口

### Phase 6 — 生态治理与权限模型预留
- Registry 索引格式
- 插件 / 主题元数据规范
- 风险标签与权限展示
